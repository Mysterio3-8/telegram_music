# СПЕКА: Мировой каталог — жанры, 4500 артистов, карточки, 30 000 треков

Задача от владельца (голосовое, 2026-07-23). Исполнение — следующая сессия
(и далее фоном: объём на недели). Закачка текущих 460 артистов уже идёт и
продолжается независимо.

## Цель

Превратить сервис из «библиотеки треков» в **музыкальный каталог**: у каждого
артиста — настоящая карточка (фото, баннер, описание, жанры, альбомы, треки),
жанры ищутся, база наполняется до ~30 000 треков за ~10 дней (3000/сутки).

## Объёмы

| Что | Сколько | Источник данных |
|---|---|---|
| Жанры | ВСЕ существующие (~300–600 с иерархией: от классики и 80-х до phonk/hyperpop) | MusicBrainz genre list + Every Noise at Once + Wikipedia |
| Русские артисты | ~1000 | Чарты Яндекс.Музыки/VK, Wikipedia, MusicBrainz (area=Russia) |
| Западные артисты | ~3000 | MusicBrainz, чарты Billboard/Spotify, Wikipedia |
| Мир (Индия, Корея, Латам…) | ~500 | MusicBrainz по странам, региональные чарты |
| Треки | ~30 000 за 10 дней | SoundCloud (приоритет) → YouTube/YT Music (фолбэк) |

## Главный источник данных — MusicBrainz (решение)

Не скрейпить YouTube/ВК руками: **MusicBrainz API** — открытый, без ключей,
отдаёт по артисту: имя/алиасы, страну, жанры-теги, описание (через Wikidata),
и — критично — **официальные ссылки** (SoundCloud, YouTube-канал, сайт).
Дополнение: Deezer API (публичный: фото артиста в хорошем качестве, топ-треки,
альбомы), iTunes Search API (жанр/альбомы), Last.fm (теги, если дадут ключ).
Скрейпинг страниц (og:image и т.п.) — только как фолбэк.

## Этапы (вертикальные срезы)

### 1. Жанры в БД
- Таблица `genres` (id, name, slug, parent_id NULL — иерархия «Электроника → Phonk → Drift Phonk»).
- Сид ~300+ жанров скриптом из MusicBrainz genre list (+ ручной топ-уровень).
- M2M: `artist_genres` (artist_id, genre_id). Трек наследует жанры артиста
  (материализовать `track_genres` не нужно — join через артиста).
- API: `GET /genres` (дерево), `GET /genres/{slug}/tracks`, поиск учитывает жанр.
- Mini App: жанры в поиске (чипы) + экран жанра.

### 2. Расширение карточки артиста
- К `artists` добавить: `banner_url`, `country` (ISO), `aliases` (json/text),
  `mbid` (MusicBrainz id — ключ для дозагрузки), `deezer_id`.
- Экран карточки — по скринам владельца из папки **`проверка/`** (появится;
  сверить и добить дизайн/функции, которых нет).
- Карточка: баннер, аватар, жанры-чипы, описание, «Топ треков», «Альбомы»,
  кнопка «Слушать всё»/«Микс по артисту».

### 3. Сбор артистов (скрипт-исследователь, Celery-очередь `research`)
- CLI/задача: по спискам стран/чартов тянет из MusicBrainz пачку артистов
  (имя, жанры, страна, ссылки, wikidata-описание) → upsert в `artists` + жанры.
- Фото: Deezer `artist.picture_xl`; баннер: Deezer/фолбэк og:image SoundCloud.
- Троттлинг: MusicBrainz требует ≤1 req/sec и User-Agent — соблюдать строго.
- Выход этапа: ~4500 карточек с жанрами и фото, БЕЗ треков — уже полезно
  (поиск по артистам/жанрам оживает).

### 4. Ссылки на музыку и закачка
- Для каждого артиста: официальный SoundCloud из MusicBrainz-relations →
  `soundcloud_sources` (пайплайн уже есть). Нет SoundCloud → официальный
  YouTube-канал → `youtube_sources` (пайплайн есть). Нет ничего → карточка
  живёт без треков, помечаем `no_source`.
- Валидация ссылки перед регистрацией (профиль существует, не пустой).
- Темп 3000 треков/сутки: 2-3 воркера `soundcloud` с ротацией прокси
  (PROXY_LIST — владелец скидывает), паузы текущие (полная после скачивания,
  короткая после DRM-скипа).
- ⚠️ **Дедуп обязателен (требование владельца)** — уже трёхслойный:
  1) `soundcloud_imported`/`youtube_imports` — обработанные ссылки не качаются повторно;
  2) отпечаток chromaprint — тот же файл под другим названием = дубль;
  3) метаданные (title+artist+длительность ±2с).
  Добавить: перед регистрацией источника — дедуп источников по нормализованному URL (есть).

### 5. Альбомы и плейлисты
- SoundCloud sets артиста → импорт как альбом: `tracks.album` + (нового не надо?
  РЕШИТЬ: отдельная таблица `albums` (id, artist_id, title, cover_url, year)
  + `tracks.album_id` — да, делаем, иначе обложек у альбомов нет).
- Плейлисты «редакции»: автогенерация по жанрам («Phonk Essentials», «Русский
  рэп 2020-х») от админ-пользователя — карточки жанров наполняются сразу.

### 6. Привязка треков к артистам
- `tracks.artist_id` (FK, nullable) — миграция; биндинг по
  `lower(trim(tracks.artist))` ↔ `artists.normalized_name + aliases`.
- Новые импорты пишут artist_id сразу (источник принадлежит артисту).
- Отчёт о несвязанных → владельцу на чистку (продолжение P4).

## Инфраструктура и риски

- **Диск**: 30к треков ≈ 200+ ГБ архива — на 20 ГБ VPS НЕ помещается.
  Треки живут по tg_file_id (файлы хранит Telegram) — это ок; архивная копия
  пишется пока есть место (гард `archive_min_free_gb`), горячее держит LRU-кэш.
  Полный архив = S3 или диск больше (SCALING.md) — решение владельца, не блокер.
- **Прокси**: обязательны для темпа 3000/сутки (владелец скидывает).
- **SQLite**: на этих объёмах записей (сотни тысяч событий) переезд на
  PostgreSQL поднимается в приоритете (шаг №1 SCALING.md).
- **MusicBrainz rate limit**: 1 req/sec → ~4500 артистов ≈ 2-3 часа чистого
  времени исследователя — ок фоном.
- **Битые/мёртвые ссылки**: помечать источник `dead`, не ретраить вечно.
