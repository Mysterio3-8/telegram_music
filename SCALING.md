# План масштабирования до 10 млн пользователей

Зафиксированный план (NEXT_SESSION P7). Не переписываем всё сразу — идём
этапами, каждый шаг дешёвый и обратимый. Текущая точка: один VPS, SQLite,
polling, всё в одном процессе uvicorn.

## Этап 0 → 10k (ближайший, обязательный)

| Шаг | Что делать | Почему |
|---|---|---|
| **PostgreSQL** | `DATABASE_URL=postgresql+asyncpg://…` в .env, `alembic upgrade head` на пустую БД, перенос данных (`pgloader` или скрипт по таблицам) | SQLite — один писатель на файл; при параллельных вебхуках/API начнутся `database is locked`. Alembic уже готов, код не меняется |
| **Индексы** | Проверить план горячих запросов: `tracks(artist)`, `tracks(title)`, `track_events(track_id, created_at)`, `user_library(user_id)` | Поиск и статистика не должны делать full scan |
| **Бэкапы** | `pg_dump` в cron + копия аудио-архива | Сейчас бэкапов нет вообще |

## Этап 10k → 100k

| Шаг | Что делать | Почему |
|---|---|---|
| **polling → webhook** | `setWebhook` на `https://keybest.cc/bot-webhook/…`, обработка в том же FastAPI-процессе (aiogram webhook mode) | Polling — один потребитель апдейтов; webhook горизонтально масштабируется воркерами uvicorn |
| **Несколько воркеров uvicorn** | `--workers 4` (или systemd-инстансы за nginx upstream) | Один процесс = одно ядро |
| **Redis-кэш чтения** | Кэшировать `/tracks` (первая страница), `/artists`, `/search/popular`, `/mix` на 1-5 мин | Эти ответы одинаковы для всех — не ходить в БД на каждый запрос |
| **Rate-limit API** | slowapi/nginx `limit_req` на `/api/` (например 20 rps/ip) | Защита от скрейперов и петель фронта |

## Этап 100k → 1M

| Шаг | Что делать | Почему |
|---|---|---|
| **Аудио мимо Python** | Стрим через nginx `X-Accel-Redirect` на файлы архива (local) или S3 presigned URL; uvicorn только проверяет подпись и отдаёт редирект | Сейчас байты трека гоняются через Python-процесс — это съест всю память/CPU первыми |
| **S3-хранилище** | Переключить `S3_*` в .env (код уже умеет), архив мигрировать `rclone` | Диск VPS кончится; S3 масштабируется и снимает IO |
| **Отдельная БД-машина** | PostgreSQL на выделенный сервер / managed | Разделить нагрузку БД и приложения |
| **Мониторинг** | node_exporter + Grafana Cloud (или UptimeRobot + journald-алерты) | На этом масштабе падения замечают пользователи, а не владелец |

## Этап 1M → 10M

| Шаг | Что делать | Почему |
|---|---|---|
| **Несколько app-серверов** | 2+ VPS за балансировщиком (nginx/haproxy), stateless-процессы | Один сервер — потолок и точка отказа |
| **Реплики PostgreSQL** | 1 primary + read-реплики; чтение (/tracks, /artists, поиск) — на реплики | Чтения — 95% нагрузки |
| **Шардинг рассылок** | Несколько бот-токенов не нужно; но рассылка 10M × 25 msg/s = 4.5 суток — резать по сегментам и слать волнами | Лимит Telegram на бота фиксированный |
| **CDN для статики Mini App** | Cloudflare перед keybest.cc | Латентность и защита от DDoS бесплатно |
| **Полнотекстовый поиск** | PostgreSQL `pg_trgm`/tsvector или Meilisearch | `ILIKE %q%` на миллионах строк умрёт |

## Что уже готово к масштабу (не трогать)

- Celery-очереди разделены по типам работ (`soundcloud`/`youtube`/`youtube_user`/`telegram_channel`) — воркеры добавляются горизонтально.
- Выдача в боте — по `tg_file_id` (Telegram CDN, наш сервер не участвует).
- Alembic-миграции: схема переезжает на PostgreSQL без правок кода.
- LRU-кэш аудио и архив в хранилище (P1) — стрим не зависит от Bot API.

## Порядок действий (когда владелец скажет «пора»)

1. PostgreSQL (полдня, риск низкий) — **это шаг №1, всё остальное после**.
2. Webhook + воркеры (полдня).
3. Redis-кэш + rate-limit (полдня).
4. X-Accel-Redirect / S3 (день).
Дальше — по росту метрик, не заранее.
